一个系统是否可靠，不只是看它多么流畅，而是看它在执行出错时能否恢复秩序。

一个仅输出文本的模型在失败时主要产生解释成本。而一个能够运行命令、写入文件、访问网络和修改代码库的模型在失败时会留下执行痕迹。目录会改变，进程会中断，配置会损坏，历史会变得难以审计。此时关键问题已不再是模型是否足够智能，而是系统是否施加了足够的约束。  -- ==体验了OpenClaw，确实有这个问题，有些更新无法回滚，就会导致变更无法预测==

对于大模型：提示决定它如何说话，驾驭（或者说是控制）决定它如何行动

**核心原则**：
1. 错误路径必须被设计为第一类路径：这是驾驭与传统助手之间一个主要区别。传统助手通常遵循“先回答，如果错了再道歉”的原则。而驾驭强调先约束，再执行；如果发生错误，会通过恢复路径而不是即兴发挥来处理。一个能道歉的系统不一定成熟。一个知道何时不应开始、何时重试、何时终止，以及如何准确报告失败的系统，则更接近成熟。
2. 验证必须是“完成”定义的一部分
3. 权限是系统的一部分，而非辅助功能
4. 上下文是一种资源，而非杂乱无章的抽屉
5. 多智能体系统依赖于角色分工，而非人工战术
6. 团队制度比个人技巧更重要

Claude Code 不是一个裸模型封装，而是一个具有上下文边界、运行时状态和行为规则的对话系统


核心的驾驭思维：
1. 是否有预算概念？
2. 是否有恢复概念？
3. 在上下文扩展后是否有救援机制？
4. 工具失败后任务能继续吗？
5. ......

Harness工程的关键原则之一是将高风险能力包装为高约束能力。能力越强，控制越精细。外部系统不会因为模型语气听起来自信就原谅糟糕的执行。  -- ==安全第一==

缓存和提示成本也很重要

一个系统是否值得被称为智能体，往往与其如何交流关系不大，而更多的取决于几轮后它是否仍清楚自己在做什么。

模型调用只是循环的一个阶段，而不是整个循环本身  -- ==智能体的作用不是只是完成一次模型调用，而是完整的循环过程==

智能体系统的核心能力是维持一个可恢复的执行循环

工具不应被视为“模型能力的扩展”，应该被视为需要有运行时管理的外部能力

工具的使用原则：
- 让模型提出行动，但不要自动授予权限；
- 在并行执行下，保留工具调度的因果关系顺序；
- 给予中断第一类语义；不要将其交给通用异常处理；
- 将高风险工具视为显式异常，而不是通用通道；
- 一个工具系统既能保护用户，又能保证运行时一致性。


## 上下文治理：内存、CLAUDE.md 和紧凑型预算机制

一旦人们可以不断向上下文塞入内容，他们很容易相信一个简单的神话：更多信息会让系统更智能。这听起来似乎合理。知道得更多应该比知道得更少更好。不幸的是，代理系统不是图书馆，模型也不是图书管理员。上下文不是一个“存储”等于“拥有”的仓库。它是一个昂贵、易通胀、自我污染的预算。  -- ==证明了一个道理：多不等于好==

**上下文是工作内存。治理的存在是为了让系统能够继续工作。**

## 错误与恢复：一个在失败后仍能持续工作的代理系统

工程中最不可信的一句话是 “在正常条件下” -- ==深以为然==

错误是主路径的一部分，恢复必须是预先设计的运行时机制

恢复机制不能变成死循环的机器。恢复系统中常见的错误既是天真的，又很昂贵：一旦一个错误被认为是“可恢复的”，就不断重试，直到可恢复变成资源灾难。

任何自动化恢复都必须可计数、速率限制且可中断。


## 多智能体协作与验证：通过分工管理不稳定性

关于多智能体协作，一个基本原则：不能把子智能体视为“另一个聊天窗口”。而是将它们视为具有明确缓存边界、状态边界、验证职责和清理责任的受管理执行进程。

多智能体系统依赖于明确的分工：研究、实现、验证和合成必须在不同的约束容器下进行，然后由协调员将其缝合回可交付成果中。


## 团队采用：将智能工具转化为可持续工作流程

真正的团队问题是把原本存在于专家头脑中的顺序转化为普通贡献者可以重复的工作流程，而不依赖于英雄式的行为。

理想的团队 CLAUDE.md 并不只是“大而全”。它应该足够稳定，以至于人们很少需要就其进行争论。它应该像基础一样运作，而不是公告板。  -- ==核心基础规则==

自动化标准流程：
- 首先标准化验证定义，就是对该技能需要有可验证的逻辑，保证结果是争取的
- 然后将重复性工作流打包成技能或命令
- 仅此之后才考虑更复杂的自动化

设计技能时尽量避免两种模型：
1. 被当作一个长提示模板
2. 被过度拔高成类似完整机构对象的东西
